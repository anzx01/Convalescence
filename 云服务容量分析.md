# 云服务容量分析报告

**分析日期**: 2026-03-07

---

### 1.2 数据库配置

## 二、容量估算

### 2.1 并发能力分析

#### Node.js API Server（2核4G）
**理论并发**:
- 单个Node.js进程: 约1000-2000个并发连接
- 使用PM2集群模式（2个进程）: 约2000-4000个并发连接
- 考虑业务逻辑复杂度: **实际支持约1000-1500并发请求**

**QPS（每秒请求数）**:
- 轻量级接口（查询）: 500-800 QPS
- 中等复杂度接口（写入+查询）: 200-400 QPS
- 复杂接口（多表关联+计算）: 100-200 QPS
- **综合平均**: 约300-500 QPS

#### PostgreSQL（1核2G）
**连接数**:
- 最大连接数: 约100-200个
- 推荐活跃连接: 50-80个
- 使用连接池后: **可支持500-1000个应用并发**

**TPS（每秒事务数）**:
- 简单查询: 1000-2000 TPS
- 写入操作: 500-1000 TPS
- 复杂查询: 200-500 TPS
- **综合平均**: 约500-800 TPS

#### Redis（1G内存）
**存储容量**:
- 1GB内存实际可用: 约800MB
- 单个用户会话数据: 约10KB
- **可存储约80,000个会话**

**QPS**:
- Redis单实例: 10,000-50,000 QPS
- **不会成为瓶颈**

### 2.2 用户规模估算

#### 场景1: 日活用户（DAU）
假设用户行为模式:
- 每个用户每天打开小程序: 3-5次
- 每次会话产生请求: 10-20个
- 每个用户每天总请求: 30-100个
- 请求分布: 高峰期（8:00-9:00, 20:00-21:00）占50%流量

**计算**:
```
高峰期QPS = DAU × 每用户请求数 × 高峰占比 / 高峰时长(秒)

假设服务器支持300 QPS:
300 = DAU × 50 × 0.5 / 3600
DAU = 300 × 3600 / (50 × 0.5)
DAU ≈ 43,200 用户

考虑20%安全余量:
实际支持 DAU ≈ 35,000 用户
```

#### 场景2: 注册用户总数
假设:
- 日活率: 60%（MVP目标）
- DAU: 35,000

**计算**:
```
注册用户总数 = DAU / 日活率
注册用户总数 = 35,000 / 0.6
注册用户总数 ≈ 58,000 用户
```

#### 场景3: 数据库存储容量
**PostgreSQL 50GB存储**:

单个患者数据量估算:
- 患者档案: 1KB
- 用药记录（1年）: 365天 × 2次 × 0.5KB = 365KB
- 健康记录（1年）: 365天 × 1KB = 365KB
- 任务记录（1年）: 365天 × 3条 × 0.3KB = 328KB
- 家属关系: 0.5KB
- **单用户年数据量**: 约1.06MB

**计算**:
```
可存储用户数 = 50GB / 1.06MB
可存储用户数 ≈ 48,000 用户（1年数据）
```

## 四、性能优化建议

### 4.1 不升级配置的优化（成本0元）

#### 代码层优化
1. **API接口优化**
   - 使用Redis缓存热点数据（用户信息、用药列表）
   - 数据库查询添加索引
   - 减少N+1查询问题
   - 使用分页加载

2. **数据库优化**
   - 为常用查询字段添加索引
   - 定期清理过期数据（>1年的记录归档）
   - 使用数据库连接池（pg-pool）
   - 慢查询优化

3. **Redis缓存策略**
   - 用户会话: 缓存24小时
   - 用药列表: 缓存1小时
   - 今日任务: 缓存到当天结束
   - 统计数据: 缓存30分钟

4. **CDN加速**
   - 静态资源（图片、图标）使用CDN
   - 小程序代码包使用CDN分发

**预期提升**: 可支持 **40,000-50,000 DAU**

### 4.2 小幅升级方案（+2,000元/年）

#### 升级配置
- 云服务器: 2核4G → **4核8G** (+1,500元/年)
- PostgreSQL: 1核2G → **2核4G** (+1,200元/年)
- Redis: 1G → **2G** (+600元/年)
- **总增加成本**: 约3,300元/年

**支持规模**:
- DAU: **60,000 - 80,000**
- 注册用户: **100,000 - 120,000**
- 并发在线: **3,000 - 4,000**

### 4.3 中等规模方案（+8,000元/年）

#### 升级配置
- 云服务器: 4核8G → **8核16G** (+4,000元/年)
- PostgreSQL: 2核4G → **4核8G** (+3,000元/年)
- Redis: 2G → **4G** (+1,000元/年)
- 增加负载均衡: (+1,000元/年)
- **总增加成本**: 约9,000元/年

**支持规模**:
- DAU: **150,000 - 200,000**
- 注册用户: **250,000 - 300,000**
- 并发在线: **8,000 - 10,000**

## 五、扩容时机建议

### 5.1 监控指标

**需要扩容的信号**:
1. **CPU使用率**: 持续 >70%
2. **内存使用率**: 持续 >80%
3. **数据库连接数**: 接近最大值的80%
4. **API响应时间**: P95 >500ms
5. **错误率**: >1%

### 5.2 扩容阶段规划

| 阶段 | DAU范围 | 配置方案 | 月成本 |
|------|---------|---------|--------|
| **MVP阶段** | 0 - 30,000 | 2核4G + 1核2G PG | 750元 |
| **成长期** | 30,000 - 80,000 | 4核8G + 2核4G PG | 1,100元 |
| **扩张期** | 80,000 - 200,000 | 8核16G + 4核8G PG | 1,800元 |
| **成熟期** | 200,000+ | 集群部署 + 读写分离 | 5,000元+ |

## 八、总结与建议

### 8.1 核心结论

**当前配置（2核4G + 1核2G PG）**:
- ✅ **完全满足MVP阶段需求**（0-30,000 DAU）
- ✅ **性价比高**（年成本8,900元）
- ✅ **可通过优化支持到50,000 DAU**
- ⚠️ **需要做好监控和扩容准备**

### 8.2 行动建议

#### 立即执行（第1周）
1. 部署监控系统（腾讯云监控）
2. 设置告警阈值（CPU>70%, 内存>80%）
3. 实施Redis缓存策略
4. 数据库添加必要索引

#### 短期优化（第1-3个月）
1. 持续监控性能指标
2. 优化慢查询
3. 实施数据归档策略
4. 准备扩容预案

#### 中期规划（第3-6个月）
1. 当DAU达到25,000时启动扩容评估
2. 准备升级到4核8G方案
3. 考虑读写分离（如需要）
4. 评估CDN效果

### 8.3 风险提示

**可能的风险**:
1. **突发流量**: 如被媒体报道，流量暴增
   - 应对: 提前准备弹性扩容方案

2. **推广超预期**: 用户增长过快
   - 应对: 设置用户增长上限，分批开放

3. **数据库瓶颈**: 写入量过大
   - 应对: 异步写入 + 消息队列

---

**报告结论**:
当前PRD规划的云服务配置（年成本8,900元）**完全满足MVP阶段需求**，可支持**30,000-35,000日活用户**，通过代码优化可扩展到**50,000日活**。建议按此配置启动项目，并在用户增长到25,000时评估升级方案。
